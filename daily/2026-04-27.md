# 2026年4月27日 - 第19课：RAG — 检索增强生成

## 今日学习主题
**理解 RAG 如何让大模型"开卷答题"，用检索外挂知识库解决幻觉和知识过时问题。**

## 为什么学这个？
上一课我们学了大模型微调——让模型适应特定领域的表达方式。但微调有个硬伤：**知识是"烧"进权重里的**，想更新知识就得重新训练。RAG 是另一条路：不修改模型，而是在生成时动态检索外部知识库，把相关资料塞进 prompt 里。

微调改的是"模型的能力"，RAG 改的是"模型能看到什么"。两者互补。

**在 AI 演进中的位置**：RAG（2020）是连接"大模型"和"Agent"的关键桥梁——它让模型第一次能"主动查资料"，为后续 Agent 自主使用工具打下基础。

## 核心概念解释

### 1. RAG 的三段流程
- **直觉**：就像开卷考试——先看题（Query），再翻书找相关段落（Retrieve），然后看着资料写答案（Generate）
- **流程**：Query → Embedding → 向量检索 → Top-K 文档 → 拼 Prompt → LLM 生成
- **代码对应**：Notebook 中的 `rag_retrieve()` + `rag_build_prompt()`

### 2. Embedding（向量化）
- **直觉**：把文字变成一串数字，语义相近的文字数字也相近
- **类比**：给每段文字一个"语义GPS坐标"，距离越近 = 意义越像
- **生产选择**：TF-IDF（简单）、Sentence-BERT（中等）、BGE/OpenAI Embedding（生产级）

### 3. Chunking（文档分块）
- **直觉**：知识库可能几十万字，不能整篇喂给模型，要切成一口能吃的小块
- **关键决策**：块太大会混入无关信息，块太小会丢失上下文
- **常见策略**：固定长度（如 512 token）、按段落、递归切分

### 4. 向量数据库
- **直觉**：专门存"向量坐标"的数据库，能在百万向量中秒级找到最近的
- **代表**：FAISS（Meta开源，本地）、Milvus（分布式）、Pinecone（云服务）
- **为什么不用 MySQL？** 传统数据库精确匹配，向量数据库做"相似度搜索"

## 关键公式直觉

| 概念 | 公式/方法 | 直觉解释 |
|------|----------|----------|
| 余弦相似度 | cos(θ) = A·B / (‖A‖×‖B‖) | 两个向量方向越一致，值越接近1 |
| TF-IDF | tf × log(N/df) | 一个词在某文档中频繁出现但在全局少见，权重高 |
| Top-K 检索 | argmax_k(cos(q, d_i)) | 找和问题最像的 K 个文档块 |
| RAG Prompt | context + query → LLM | 把检索结果当"参考资料"贴在问题前面 |

## 动手实践要点
- Notebook 用 TF-IDF 做简化演示，理解核心流程
- 生产环境用 Sentence-Transformer 或 OpenAI Embedding 做向量化
- 观察：不同 query 检索出的 top-k 文档是否合理
- 注意：chunk 大小、top-k 值是最需要调的两个参数

## 常见问题

**Q: RAG 能完全替代微调吗？**
A: 不能。RAG 解决"知识注入"，微调解决"能力/风格适配"。比如让模型用法律文书风格回答，微调更合适；让模型能回答公司内部最新政策，RAG 更合适。最好的做法是两者结合。

**Q: 如果检索到了不相关的文档怎么办？**
A: 这是 RAG 的核心挑战。解决方案包括：用 Reranker（Cross-Encoder）精排、提高 Embedding 模型质量、优化 Chunking 策略、在 Prompt 中明确告知模型"参考资料的可靠性不同"。

## 本课总结
1. RAG = 检索 + 增强 + 生成，本质是让模型"先查资料再回答"
2. 核心工程组件：Embedding 模型、向量数据库、Chunking 策略
3. RAG 解决知识时效性和幻觉问题，但检索质量是关键瓶颈
4. RAG 和微调互补，不是二选一
